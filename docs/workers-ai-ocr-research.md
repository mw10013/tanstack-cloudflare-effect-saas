# Workers AI OCR Research

## Summary

Workers AI has no dedicated OCR model. OCR-like capability comes from two approaches:

1. **`env.AI.toMarkdown()` (Markdown Conversion)** — high-level utility, easiest path
2. **Direct vision model invocation** — lower-level, more control over prompts

## Approach 1: `toMarkdown` for Image → Markdown → LLM Extraction

### How It Works for Images

The `toMarkdown` pipeline for images:

1. Resizes image if > 1280×720
2. Runs **`@cf/facebook/detr-resnet-50`** for object detection
3. Feeds detected objects + image to **`@cf/google/gemma-3-12b-it`** (12B multimodal) for image-to-text

The output is a **natural-language summary/description** generated by Gemma 3, not verbatim text extraction. For PDFs, `toMarkdown` parses the actual text layer (`StructTree`) and returns exact text. For images, it feeds the image to a vision LLM and gets back whatever the model writes.

**Concretely for an invoice image with line items:** The model would output something like _"The image shows an invoice from Cloudflare, Inc. dated March 4, 2026. It lists several line items including Durable Objects Storage, Browser Rendering hours, and Workers Paid at $5.00. The total amount due is $5.39 USD."_ — a prose summary, not the raw text. It will paraphrase, omit details, round numbers, and miss items. It is **not OCR**. There are no coordinates, no bounding boxes, no verbatim text positions.

So this is not suitable for feeding into another llm for invoice extraction, right? Why are you recommending this as an option? If it doesn't make sense for our use case, then note that and why and move on.

**Workers AI has no true OCR model.** No model returns text with coordinates/bounding boxes (like Tesseract or AWS Textract). The `@cf/facebook/detr-resnet-50` object detection model returns bounding boxes for objects (cats, people, etc.) but not for text regions.

### Image → toMarkdown → LLM Pipeline (Matches Existing Pattern)

This project already does: **PDF → `toMarkdown` → `@cf/openai/gpt-oss-120b`** for structured invoice extraction (see `src/lib/invoice-extraction.ts`). The same two-step pipeline could work for images, but the quality of step 1 is much worse for images than PDFs because you're getting a lossy description rather than exact text.

```typescript
// Step 1: Image → markdown via toMarkdown (lossy — description, not OCR)
const markdownResult = await env.AI.toMarkdown({
  name: "invoice.jpeg",
  blob: new Blob([imageBuffer], { type: "image/jpeg" }),
});
// markdownResult.data => prose description like "The image shows an invoice..."

// Step 2: Feed markdown to LLM for structured extraction (existing pattern)
const extracted = await runInvoiceExtractionViaGateway({
  accountId, gatewayId, workersAiApiToken, aiGatewayToken,
  markdown: markdownResult.data,
});
```

**Problem:** The description from step 1 will lose line item details, exact amounts, addresses, etc. The 120B model in step 2 can only extract what the 12B model chose to mention in its description.

### Binding Usage

```typescript
const result = await env.AI.toMarkdown({
  name: "invoice.jpeg",
  blob: new Blob([imageBuffer], { type: "image/jpeg" }),
});
// result.data => markdown string
// result.tokens => estimated token count
```

### REST API Usage

```bash
curl https://api.cloudflare.com/client/v4/accounts/{ACCOUNT_ID}/ai/tomarkdown \
  -X POST \
  -H 'Authorization: Bearer {API_TOKEN}' \
  -F "files=@invoice.jpeg"
```

### Conversion Options

```typescript
await env.AI.toMarkdown(files, {
  conversionOptions: {
    image: { descriptionLanguage: "en" }, // en | it | de | es | fr | pt
  },
});
```

`descriptionLanguage` adds a directive to the prompt sent to `@cf/google/gemma-3-12b-it` telling it to output the image description in the specified language. It only affects the language of the AI-generated description — it does not do translation of text found in the image. Best-effort: the model may still output in a different language.

For this project: yes, use `"en"` or just omit it (English is the default).

### Costs

`toMarkdown` is free for most formats. For images specifically, it runs **two Workers AI models** under the hood:
- `@cf/facebook/detr-resnet-50` (object detection)
- `@cf/google/gemma-3-12b-it` ($0.345/M input tokens, $0.556/M output tokens)

These count against your Workers AI neuron allocation and will incur costs beyond the free 10,000 neurons/day.

## Approach 2: Direct Vision Models

### Available Vision/Image-to-Text Models

| Model | Params | JSON Mode | Notes |
|---|---|---|---|
| **`@cf/google/gemma-3-12b-it`** | 12B | No | Used by `toMarkdown` internally. Multimodal. |
| **`@cf/meta/llama-3.2-11b-vision-instruct`** | 11B | ✅ Yes | Vision model with JSON mode + structured output. |
| `@cf/llava-hf/llava-1.5-7b-hf` | 7B | No | Older, smaller. |

### Why `toMarkdown` Uses Gemma 3 Instead of Llama 3.2 Vision

`toMarkdown` uses `@cf/google/gemma-3-12b-it` because it's a general-purpose pipeline that needs to work without user prerequisites. Llama 3.2 Vision requires explicit Meta license acceptance per account, which would break the zero-config nature of `toMarkdown`. Gemma 3 has no such license gate.

### Best for Direct Image Text Extraction: `@cf/meta/llama-3.2-11b-vision-instruct`

This is the best option on Workers AI for getting raw text out of an image. **Not true OCR** (no coordinates/bounding boxes — Workers AI simply doesn't have that), but you can prompt it to dump all visible text verbatim rather than summarize.

**Don't use JSON mode with the invoice extraction schema here.** An 11B model can't reliably map a complex invoice into a structured schema with line items, addresses, etc. Instead, use it as a **text extraction step** — prompt it to output all text it sees, then feed that text to the 120B model (`gpt-oss-120b`) for structured extraction.

```typescript
// Use as raw text extractor (no JSON mode, no schema)
const response = await env.AI.run("@cf/meta/llama-3.2-11b-vision-instruct", {
  messages: [
    {
      role: "user",
      content: "Extract all text from this image exactly as it appears. Preserve the layout, line breaks, and formatting. Do not summarize or paraphrase.",
    },
  ],
  image: base64ImageData, // "data:image/png;base64,..." or raw base64
  max_tokens: 4096,
});
// response.response => raw text from the image

// Then feed to gpt-oss-120b for structured extraction (existing pipeline)
const extracted = await runInvoiceExtractionViaGateway({
  accountId, gatewayId, workersAiApiToken, aiGatewayToken,
  markdown: response.response,
});
```

This is better than `toMarkdown` for images because you control the prompt — you can tell the model to extract text verbatim instead of summarizing. Still an 11B vision model so accuracy on dense tables with small text may vary.

This sounds pretty shitty. How can we really expect to get invoice data accurately?

### Pricing Comparison

| Model | Input | Output |
|---|---|---|
| `@cf/google/gemma-3-12b-it` | $0.345/M tokens | $0.556/M tokens |
| `@cf/meta/llama-3.2-11b-vision-instruct` | $0.049/M tokens | $0.676/M tokens |
| `@cf/openai/gpt-oss-120b` (current extraction model) | $0.350/M tokens | $0.750/M tokens |

Llama 3.2 Vision is **7× cheaper on input** than Gemma 3.

### Rate Limits

- Image-to-Text: 720 requests/minute

## Recommendation

### Option A (Recommended): Image → `llama-3.2-11b-vision-instruct` (text extraction) → `gpt-oss-120b` (structured extraction)

Best accuracy. Use Llama 3.2 Vision as a text extractor with a "dump all text verbatim" prompt, then feed the output to the existing `runInvoiceExtractionViaGateway` pipeline with `gpt-oss-120b`. You control the extraction prompt (unlike `toMarkdown` which summarizes), and the 120B model handles the complex schema mapping. Two model calls but each doing what it's good at.

### Option B: Image → `toMarkdown` → `gpt-oss-120b` (Simpler, Lower Quality)

Uses existing `toMarkdown` pipeline. Easier to implement (no Meta license needed) but `toMarkdown` summarizes instead of extracting text verbatim. Will lose line item details.

### Option C: Image → `llama-3.2-11b-vision-instruct` with JSON Mode (Single-Step, Cheapest, Least Reliable)

Single model call, cheapest. But asking an 11B model to both read an image AND map it to a complex invoice schema is too much. Will fail on dense invoices.

## Prerequisites

### AI Binding

```toml
[ai]
binding = "AI"
```

### Meta License for Llama 3.2 Vision

Required **once per Cloudflare account** before first use. Send a single API request:

```bash
curl https://api.cloudflare.com/client/v4/accounts/$CLOUDFLARE_ACCOUNT_ID/ai/run/@cf/meta/llama-3.2-11b-vision-instruct \
  -X POST \
  -H "Authorization: Bearer $CLOUDFLARE_AUTH_TOKEN" \
  -d '{ "prompt": "agree" }'
```

This is a one-time acknowledgment per Cloudflare account. After this, the model is immediately available — no approval wait. You run this once from your terminal before deploying. It doesn't need to be in your Worker code. Subsequent deployments and all Workers on the same account can use the model without repeating this.
